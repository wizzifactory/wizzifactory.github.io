Introduction

The final goal of the Wizzi factory is the model driven generation of Software Artifacts.

Artifact generators do the final job, emitting code units, scripts and documents targeting the programming languages and technologies of a software package.

The main job of the factory is the production of rich semantic models, trasforming conceptual specifications of requirements into specific domain syntax trees.

The Wizzi factory consist of an organic set of tools for cascading the process from conceptual specifications to Software Artifacts.

Conceptual specifications and their transformations are defined in textual documents named IttfDocuments.

A pipeline of transformations is applied to IttfDocuments giving as a result a typed WizziModelInstance validated against a WizziSchema. Wizzi schemas define the semantics of a WizziModel.

At the end Artifact generators transform WizziModelInstances into Software Artifacts.

The Wizzi factory has two main processes:

  • A Model type generation process:

    • from a WizziSchema to a WizziModelType and a WizziModelLoader.

  • An Artifact generation process:

    • from one or more IttfDocuments to an IttfModel by an IttfLoader;
    • to a WizziModelInstance by a WizziModelLoader;
    • to a Software artifact by an Artifact generator.

The two processes have the same structure because WizziModelTypes and WizziModelLoaders are themselves Software artifacts generated by an Artifact generation process and WizziSchemas are modeled using IttfDocuments.

There is a circularity that requires a bootstrap step. It consists of a manually crafted WizziModelType describing a WizziMetaSchema that is a WizziSchema of type 'wizzischema'. WizziModelInstances of type 'wizzischema' are inputs to the Model type generation process that generates WizziModelTypes and WizziModelLoaders for the domains of interest.

Concepts

The modeling system of the Wizzi Factory

Wizzi models are based on the concepts of element and relation. An element is a classifier and a relation is a container relationship between elements.

The modeling system is three layered: a meta model, the WizziMetaSchema, models a WizziSchema, which in turn models a WizziModel type.

The WizziMetaSchema is designed to model WizziSchemas and WizziModels that can be represented by a synthax three structure and can be easily mapped to a javascript object.

In a WizziModel:

  • An element may have one super element (abstraction) and derived elements (specializations), may be abstract or concrete and may have a name and attributes. The schema says if the name is required and which are valid attributes.
  • Relationships are of two types: one-to-many (default) and one-to-one. They do not have attributes. Both imply ownership of contained (child) elements: in case one-to-many an array of child elements and in case of one-to-one a single child element.
  • An attribute has a name and a value of a primitive type.
  • Elements, relationships and attributes may have constraints.

This tree structure can be mapped to a javascript pseudo object oriented model, where:

  • Elements are pseudo-class functions.
  • Attributes are variables, of pseudo-class instances, with a primitive type.
  • One-to-many relationships are array values on the related instance.
  • One-to-one relationships are object values, on the related instance.
  • Attributes and relationships are inherited from super elements.

The notation for elements and attributes may be abbreviated. The schema can assign a short mnemonic tag to elements and attributes.

Actually a WizziModel type IS a javascript object model of pseudo-class functions, that is auto-generated from a WizziSchema.

A WizziModel instance can perform actions. A WizziSchema can declare element methods that will be inserted in the generated javascript WizziModel and will have access to reads and manipulations of the model instance.

Two methods: wzInitialize and wzVerify, are autogenerated and performs initialization and validation actions implict in the WizziSchema and its constraints.

Users may define new methods and/or add code lines to the wzInitialize and wzVerify methods.

Templateability

WizziModel instances drive the artifact generators of the Wizzi Factory, but they can't be crafted directly; they are instances created in memory by WizziModel loaders that parse source IttfDocuments.

IttfDocuments, written using the ITTF notation, are the concrete building blocks of models and templates of the Wizzi Factory.

Every IttfDocument is a full template: all its nodes are templeatable; even the root of a primary IttfDocument may be a replaceable node or a node with a template expression in its value.

The notation of an IttfDocument is quite simple: a syntax tree where each node is a name value pair separated by a whitechar. The node name (or tag) is the symbol of the syntax node and may represent a structure or an action. The value of the node contains the name of the structure or the operands of the action.

Node values and operands are not formally specified by the WizziSchema. But the schema can:

  • Express their meaning in comments.
  • Validate them in the wzVerify method.
  • Manipulate them in other element/node methods.

Build up of WizziModel instances

The buildup and loading of a model instance from one or many IttfDocuments is the most powerful process of the Wizzi Factory.

An IttfDocument may contain two kind of nodes:

  • Domain nodes, that are name/value pairs which semantics is described by a WizziSchema.
  • Control nodes, alias IttfStatement nodes, that are processing and conditional statements that, once the loading is triggered, drive the buildup of the model instance.

Only domain nodes are templatable.

IttfStatements nodes allow:

  • Inclusion or mix up of other Ittf documents.
  • Conditional inclusion/exclusion of node branches using context values.
  • Creation or modification of context values by code written inside the Ittf document.

The mixup af IttfDocuments can be recursive: a mixin can in turn mixup other mixins. The node with the IttfStatemnts that calls a mixin inclusion may be the root node of on IttfDocument, allowing a partial branch to mixup its container root and to be appended to a declared node (see the IttfStatements: $append and $hook).

One can pass any object as a template context object to the WizziModel loader.

The final step of the build up is the template evaluation. During evaluation the syntax tree of the source IttfDocuments is rebuilt into an in memory tree structure, the IttfModel, that contains only domain nodes and may be verified and loaded into a WizziModel instance.

WizziModels as Syntax Trees

WizziModel types are generated in javascript implementing a logical tree view of the model:

  • Elements are nodes.
  • Container elements are parents (not super!) to attributes and related elements.
  • There must be only one root element (an element without a parent).

Being the generation of software artifacts its main goal the Wizzi Factory is interested both in models that describe a domain of interest than in model templates that can declare instructions for a target programming language.

The tree structure of a WizziModel allows its use as a syntax tree. Element and attribute nodes can be modeled by WizziSchemas as symbols/operands pair. Grammars may be simplified/trivialized, using stereotypes instead of valid symbols.

Elastic synthax tree

The statements of a programming language (PL) are syntactic sugar, the best as possible human readable, that must be parsed into an abstract synthax tree (AST) in order to produce the machine instructions to be executed.

The grammar of a PL is a transformation rule from syntactic sugar to AST. An AST is less human readable but more powerful of a PL. It is made of nodes that express the semantics of instructions and can be traversed to extract relations between nodes. Compilers use the concept of Abstract semantic graph (ASG) for trasformations of ASTs that may contain shared subterms (also known as "common subexpressions").

ASTs and their traversals and trasformations are a powerful tool for compiling machine instructions. Why this power can't be used in model driven generation of software artifacts, which are the inputs to compilers?

Well, obviously software factories try just to do that. Every modeling methodology is, in this respect, a kind of sintactic sugar around implicit ASTs that artifact generators traverse and trasform in the best intelligent way they can do.

But why not to use explicit ASTs as models of the software factory. ASTs transformations could be executed in every step of the factory life-cycle: models design, model instances construction and models processing in the final artifact generation step.

That is the idea on which is founded the Wizzi Factory. Everything: schemas, models, templates and generators, became ASTs, with plenty of functionality to traverse, transform, build the node branches of the tree.

That means that we go back to early times when programs where punched in machine code?

Or may an AST be elastic and contain a mix of formal tree structures and syntactic sugar, to have a balance between the power of the AST and the readability of a more natural language.

A compiler that should digest an "elastic AST" could simply spit it.

But an artifact generator that is created from the same designer/programmer/team that has created the "elastic AST" schema and model instances and their transformations can well digest it without too much hassle.

The Wizzi Factory is built on the concepts of "Elastic syntax trees" and "Code model".

Wizzi code model

A WizziCodeModel is a WizziModel modeled by a WizziSchema that targets a programming languace (PL).

Examples:

  • javascript
  • html
  • cascading style sheet

It can be viewd as a profile of a PL Grammar that can be implemented partially and with some or all its symbols stereotyped.

Is up to the developer/team that models or codes an artifact generator to establish how much 'elasticity' must have the syntax tree modeled by the schema of a WizziCodeModel respect to the target PL Grammar. Is the artifact generator that has the responsability to generate a valid, compilable software artifact.

Software component responsabilty

A software component is a centre of responsability in the wider economy of a software project. It must have its roles and competences assigned by the project managers to optimize the project performance.

A declarative model has its responsability assigned by its schema. It must express clearly the semantics of the schema. It has not to be perfect or complex per se.

Target technology

The Wizzi factory is technology agnostic and try to separate the 'what' from the 'how'.

Each Software artifact targets one technology.

Concept schemas

A concept schema is an high-level abstraction schema that do not target a specific target technology.

It can be used as a data context to enrich other more specific Wizzi schemas before or during artifact generation.

Examples:

  • Type system
  • Layout
  • Path management

Factory items

Ittf documents

Ittf stays for Indented Text Tree Format, the notation format of the Wizzi Factory modeling system.

An Ittf document is a syntax tree, codified in a text document, where child nodes are indented to their parent. This is an Ittf document:

  1. html 
  2.     body 
  3.         ul 
  4.             li 
  5.                 a The Wizzi Factory
  6.                     href https://wizzifactory.github.io

There are primary, mixin and include Ittf documents. This is a mixin: 

  1. li 
  2.     $params text, href
  3.     a ${text}
  4.         href ${href}

To a mixin you can pass arguments. Instead an include is simply inserted as is. This could be an include: 

  1. div License MIT
  2.     span copy Stefano Bassoli.

It may contain two kind of nodes:

  • Object nodes. Name/value pairs which semantics is described by a Wizzi schema.
  • Control nodes, alias Ittf statement nodes. Processing and conditional statements that, once the loading is triggered, drive the document into an Ittf model, a syntax tree with only node objects.

Every IttfDocument of the Wizzi Factory is a full template: all its nodes are templeatable elements; even the root of a primary IttfDocument may be a replaceable node or a node with a template expression in its value.

An Ittf document can include or mix up other Ittf documents. One can pass any object as a context to the Ittf model loader, can conditionaly include/exclude node branches using context values and can also create context values by code inside an Ittf document.

The final result only, alias the Ittf model, containig object nodes only, must be validated by a Wizzi schema to became a Wizzi model instance.

Not every Itff document should became a Wizzi model. An Ittf model once built can be persisted again to a new, more complex Itff document, so you can chain as many transformations as you need and load only final Ittf models into Wizzi model instances.

see the IttfStatements API

see IttfDocuments examples

Ittf statement

An IttfStatement is a control node of an IttfDocument, that is recognized by name by the IttfModelLoader.

Are IttfStatement:

  • include or mixin call instructions
    • $include
    • ( [, [ … ] ] )
    • $params [, [ … ] ] )
  • template control statements
    • $if
    • $elif
    • $else
    • $foreach in
  • append and hook statements
    • $append
    • $hook
  • context manipulation code
    • $
    • $global
  • multiline value content
    • $.
  • comments

Ittf models

The Ittf model is the result of the loading of an Ittf document.

It contains semantic nodes only, that must be conformant to a Wizzi schema, and is loaded into a Wizzi model instance.

see the IttfModel API

Ittf model loaders

The IttfModelLoader is the transformation engine of the Wizzi factory. It parses IttfDocuments and processes It processes the Ittf statements coded in Ittf documents and composes, mixups, evaluates the tree nodes of the documents producing an Ittf model.

see the Ittf model load process description

Wizzi schemas

A schema to which an Ittf model must be conformant in order to be loaded into a Wizzi model instance.

Wizzi models and Wizzi model loaders are artifacts generated transforming a Wizzi schema.

A Wizzi schema is simply a Wizzi model instance of a meta schema of type 'wizzischema' and is modeled using an Ittf document, just like every Wizzi factory model:

  1. wizzischema rdbms
  2.     e catalog
  3.         r table/s
  4.     e table
  5.         r column/s
  6.         r index/es
  7.     e column
  8.         a type
  9.         a caption
  10.         a defaultValue
  11.             tag default
  12.     e index
  13.         a primary
  14.             type boolean
  15.         a unique
  16.             type boolean
  17.         r column/s

A Wizzi schema may target every kind of technology: data, ui, solution, code, etc ...

Wizzi models

A Wizzi model is a typed Document Object Model acting as a data context in an Artifact generation process.

Wizzi model(s) and their loader(s) are generated from a Wizzi schema.

Instances of a Wizzi model are created by Wizzi model loader(s) and populated with the semantic nodes of an Ittf model

see the WizziModel API

Wizzi model loaders

A WizziModelLoader is a javascript module generated from a WizziSchema during a ModelTypeGenerationProcess.

A WizziModelLoader executes the loading of an IttfDocument into a WizziModelInstance of its type.

The loading process consist of:

  • Template context models loading:

    • The recursive execution of this same loading process for each context model passed as an argument to the WizziModelLoader;
    • TemplateContext emission (callback).

    An IttfDocument load execution:

    • IttfDocument lexing and parsing;
    • IttfDocument inclusions;
    • IttfDocument mixups (or mixin resolutions);
    • IttfModel node branch appends and hooks;
    • IttfModel template evaluation (against the model instances of the TemplateContext) and rebuild;
    • IttfModel emission (callback).

    An IttfModel load execution

    • WizziModelInstance instantiation from a WizziModelType;
    • WizziModelInstance loading from the IttfModel;
    • WizziModelInstance initialize;
    • WizziModelInstance verify;
    • WizziModelInstance emission (callback).

Wizzi model instances

A WizziModelInstance is an instance of a WizziModelType populated with an IttfModel conformant to its WizziSchema.

It can act as a context object in the template evaluation of other WizziModelInstances. May be used as a context by CodeWriteArtifactGenerators. May be traversed and manipulated by WizziModelTransforemers or by WizziSchema methods.

Wizzi model transformers

An IttfDocument has powerful statements for composing rich models mixing context data, but you may need to preprocess your WizziModelInstance before Artifact generation.

A Wizzi model transformer is a javascript program that receives in input a WizziModelInstance and an object context, and returns a transformed WizziModelInstance or a POJO object.

To be exposed to the ProductionManager of the Wizzi Factory, WizziModelTransformers must be packaged respecting a conventional folder structure and an interface.

see WizziModelTransformerExample

see WizziModelTransformerPackage

see WizziModelTransformerInterface

Artifact generators

An Artifact generator transforms one or more WizziModelInstances into a Software artifact.

It can be:

  • a full Javascript program, that uses a 'code write' process to output an artifact;
  • an IttfDocument that act as a template and will be loaded into a WizziCodeModel, from which the artifact will be generated.

To be exposed to the ProductionManager of the Wizzi Factory, artifacts must be packaged respecting a conventional folder structure and an interface.

see CodeWriteArtifact

Ittf statements

$

One can declare context values inside an Ittf document and/or manipulate them using Ittf expressions. Context manipulation statements may be single or multi node. The single node format is:

  1.  
  2.     $ var counter = 0
  3.

The multi node format is:

  1.  
  2.     $ 
  3.         var x = {};
  4.         x.value = 10;
  5.

$global

Context values declared inside an Ittf document using the $ statement are local to the document. To make them global you must declare them using the $global statement insted of $. The syntax is the same. Single node ...

  1.  
  2.     $global var counter = 0
  3.

... and multi node ...

  1.  
  2.     $global 
  3.         var x = {};
  4.         x.value = 10;
  5.

$if

The $if, $elif and $else statements allow conditional inclunsion/exclusion of nodes. The format is:

  1.  
  2.     $if ittf-expression
  3.         .. child nodes
  4.     $elif ittf-expression
  5.         .. child nodes
  6.     $else 
  7.         .. child nodes
  8.

$if and $elif child nodes will be included in the Ittf model if ittf-expression is true, while $else child nodes will be included if ittf-expression is false.

The $foreach statements repeats the processing of child nodes for each element in a context value of type array. The format is:

mixin call

A mixin call is an Ittf statement that the loader recognizes because a node name ends with an open paren. You can declare single node mixin calls:

  1.  
  2.     ui/dialog( title, &data )
  3.

or multi node calls: 

  1.  
  2.     ui/control( 
  3.         $arg id name
  4.         $arg type input
  5.         $arg label Your name
  6.

Arguments in a single node call must be comma separated. The closing paren is optional in both cases.
