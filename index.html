Introduction

The final goal of the Wizzi factory is the model driven generation of Software Artifacts.

Artifact generators do the final job, emitting code units, scripts and documents targeting the technologies of an application or system.

The big job is the production of rich semantic models trasforming conceptual specifications of requirements.

The Wizzi factory consist of an organic set of tools for cascading the process from conceptual specifications to Software Artifacts.

Conceptual specifications are stored in textual documents named Ittf documents.

A pipeline of transformations is applied to Ittf documents giving a typed Wizzi model instance validated against a Wizzi schema. Wizzi schemas define the semantics of a Wizzi model.

At the end Artifact generators transform Wizzi model instances into Software Artifacts.

The Wizzi factory has two main processes:

  • A Schema definition process: from a Wizzi schema to a Wizzi model and a Wizzi model loader
  • An Artifact generation process: from Ittf documents to an Ittf model by an Ittf loader, to a Wizzi model instance by a Wizzi model loader, to a Software artifact by an Artifact generator.

The two processe have the same structure because Wizzi models and Wizzi model loaders are themselves Software artifacts generated by an Artifact generation process and Wizzi schemas are defined in Ittf documents.

There is a circularity that requires a bootstrap step. It consists of a manually crafted Wizzi model describing the Wizzi schema of type 'wizzischema'. Wizzi model instances of type 'wizzischema' are inputs to the Schema generation process that generates Wizzi models and Wizzi model loaders for the domains of interest.

Objects

Ittf documents

(Indented text tree format)

The source format of all Wizzi factory models.

An Ittf document is a logical tree, codified in a text document, where child nodes are indented.

It may contain two kind of nodes:

  • Semantic nodes. Name/value pairs which semantics is described by a Wizzi schema.
  • Ittf statement nodes. Processing and conditional statements that, once the loading is triggered. drive the document into an Ittf model, alias a full semantic tree object.

An Ittf document is schema agnostic. Semantic nodes can contain expressions. It works as a template and can include or mix up other Ittf documents. You can pass any object as a context to the Ittf model loader. You can conditionaly include/exclude node branches using context values. You can also create context values by code inside an Ittf document.

The final result only, alias the Ittf model, must be validated by a Wizzi schema to became a Wizzi model instance.

Not every Itff document should became a Wizzi model. An Ittf model once built can be persisted again to a new, more complex Itff document, so you can chain as many transformations as you need and load only final Ittf models into Wizzi model instances.

see the IttfStatements API see IttfDocuments examples

Ittf statements

Is a node of an Ittf document, recognized by name by the Ittf model loader, that contains a loading instruction. Ittf document nodes are name/value pairs.

Examples are:

  • $include
  • $if
  • $elif
  • $else
  • $foreach in
  • ( [, [ … ] ] )
  • $params [, [ … ] ] )
  • $append
  • $hook
  • etc …
see the IttfStatements API

Ittf models

The Ittf model is the result of the loading of an Ittf document.

It contains semantic nodes only, that must be conformant to a Wizzi schema, and is loaded into a Wizzi model instance.

see the IttfModel API

Ittf model loaders

The Ittf model loader is the transformation engine of the Wizzi factory. It processes the Ittf statements coded in Ittf documents and composes, mixups, evaluates the tree nodes of the documents producing an Ittf model.

see the Ittf model load process description

Wizzi schemas

A schema to which an Ittf model must be conformant in order to be loaded into a Wizzi model instance.

Wizzi models and Wizzi model loaders are artifacts generated transforming a Wizzi schema.

A Wizzi schema is simply a Wizzi model instance of a meta schema of type ‘wizzischema’ and is modeled using an Ittf document, just like every Wizzi factory model.

A Wizzi schema may target every kind of domain: DataModeling, UIModeling, SolutionModeling, CodeModeling, etc ...

Wizzi models

A Wizzi model is a typed Document Object Model acting as a data context in an Artifact generation process.

Wizzi model(s) and their loader(s) are generated from a Wizzi schema.

Instances of a Wizzi model are created by Wizzi model loader(s) and populated with the semantic nodes of an Ittf model

see the WizziModel API

Wizzi model loaders

Is built from a Wizzi schema.

A Wizzi model loader instantiate a Wizzi model instance, verifies an Ittf model against its Wizzi schema and populate the Wizzi model instance with the Ittf model.

Wizzi model instances

Wizzi model instances are instances of a Wizzi model populated with an Ittf model conformant to its Wizzi schema.

Wizzi model transformers

An Ittf document has powerful statements for composing rich models mixing context data, but you may need to preprocess your Wizzi model before Artifact generation.

A Wizzi model transformer is a javascript program that you can write implementing a simple interface.

Artifact generators

An Artifact generator transforms one or more Wizzi model instances into a Software artifact.

It can be a Javascript program or an Ittf document that will be loaded into a Wizzi model that targets a Code modeling specific domain.

Concepts

Target technology

The Wizzi factory is technology agnostic and try to separate the 'what' from the 'how'.

Each Software artifact targets one technology.

Concept schemas

A concept schema is an high-level abstraction schema that do not target a specific target technology.

It can be used as a data context to enrich other more specific Wizzi schemas before or during artifact generation.

Examples:

  • Type system
  • Layout
  • Path management

Code schemas

A code schema is a specific domain Wizzi schema that targets a programming languace.

Examples:

  • javascript (js)
  • html
  • cascading style sheet (css)
